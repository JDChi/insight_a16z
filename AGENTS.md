# AGENTS.md

## 项目定位

这是一个部署在 `Cloudflare` 上的 `a16z AI 中文洞察站`。

目标不是做英文原文翻译站，而是：

- 收集 `a16z` 官网 `Articles` 与 `Investment News`
- 用中文输出结构化分析结果
- 保留专题、周报等聚合能力作为后续演进空间
- 让前台直接消费已发布内容

中文标题代表本站的洞察标题，不是原文直译。英文原标题只作为来源信息保留。

## 当前技术结构

- `apps/web`: `Astro` 前台内容站和最小后台
- `apps/api`: `Hono + Cloudflare Workers` API、采集、分析、任务流
- `packages/core`: 共享 schema、类型、fixture
- `migrations`: `D1` 数据库初始化脚本

## 数据源边界

只允许采集：

- `a16z` 官网
- `Articles`
- `Investment News`

不要采集：

- `Podcast`
- `Video`
- 栏目页、导航页、落地页
- 其他媒体、基金或第三方网站

## 内容策略

- 页面主标题使用中文洞察标题
- 原文英文标题保留为来源信息
- 页面正文是本站分析结果，不是原文全文翻译
- 前端可见 UI 必须保持中文

## 前台信息架构与 UI 约定

当前公开前台只展示三个一级入口：

- `文章`: 只展示来源为 `Articles` 的中文分析
- `投资动态`: 只展示来源为 `Investment News` 的中文分析
- `归档`: 展示全部已发布内容，作为补充浏览入口

重要规则：

- `文章` 列表不能混入 `Investment News`
- `投资动态` 列表不能混入普通 `Articles`
- `专题`、`周报` 暂时不要出现在公开导航、首页模块、详情页关联区或其他前台可见入口
- 不要把 `专题` 当作和 `文章`、`投资动态` 并列的公开栏目，除非后续重新明确产品定义
- 不要把 `周报` 当作首页主叙事或一级栏目，除非后续重新明确它的价值
- 首页按上下结构展示：先 `最新文章`，后 `最新投资动态`
- 首页 `最新文章` 最多展示 4 篇，`最新投资动态` 最多展示 3 篇
- 主列表页和归档页当前不提供搜索框；内容量和检索需求明确增加前，不要添加“看起来有用但实际价值不高”的搜索 UI

日期展示约定：

- 列表卡片保留发布时间，方便用户判断新旧
- 详情页标题区不展示发布时间，发布时间放在 `来源信息` 区
- 前台日期按北京时间展示为中文年月日格式

卡片布局约定：

- 两列卡片不要互相拉伸高度
- 摘要最多展示 5 行，超出后截断并做渐隐处理
- 不要改成瀑布流布局，除非后续明确改变列表阅读方式

## 发布与状态流转

文章状态以数据库为准。

当前约定状态：

- `ingested`: 已抓取入库，待分析
- `processing`: 正在分析
- `published`: 已发布到前台
- `failed`: 技术性失败，需人工排查

重要规则：

- 不再走人工审核前置流程
- 分析成功后直接发布
- 如果 AI 输出不合理，不要发布
- 不合理输出应回退到 `ingested`，等待后续重试
- 不要用劣质 fallback 结果冒充正常分析结果

## 队列与恢复策略

当前采用 `cron` 驱动的可恢复批处理模式，而不是内存长跑队列。

正确行为：

- 抓取先落库
- 后续由 `cron` 周期性扫描 `ingested`
- 每次只处理一小批
- 进程重启后，不依赖内存恢复
- 下次 `cron` 继续从数据库状态接着跑

不要重新引入：

- 单进程内存长跑队列
- 重启即丢状态的任务执行器

## 去重与质量约束

- 以 `source_url` 做文章级去重
- 中文标题不能与现有已发布标题撞车
- 如果生成结果与现有标题重复，应视为不合理输出并回退
- 不要为了“不断队列”而批量生成模板化、同质化标题

## 本地开发约定

- 本地敏感配置使用 `apps/api/.dev.vars`
- `.dev.vars`、`.env` 不允许提交
- `wrangler.toml` 可能包含本地 Cloudflare 绑定信息，修改前先确认是否应提交

本地常见模式：

- `apps/web` 可用 fixture 模式
- 联调时让前端走本地 API
- 真实模型 key 仅通过环境变量注入
- 如果已启用 Cloudflare 的 Git 自动部署，任何推到生产分支的提交都会触发线上构建，提交前要明确这是否符合预期

## 测试约定

改动以下行为时，必须补或更新测试：

- 抓取规则
- AI 分析输出修复逻辑
- 状态流转
- 队列处理
- 后台接口语义

优先覆盖：

- 单元测试：解析、标题、防重、输出校验
- 集成测试：状态流转、发布、后台接口
- E2E：前台和后台核心页面

## Commit 规范

从现在开始，commit message 使用统一前缀：

- `feat:` 新功能或新增能力
- `fix:` 缺陷修复或行为纠正
- `refactor:` 不改变外部行为的重构
- `docs:` 文档更新
- `test:` 测试补充或调整
- `chore:` 杂项维护

提交信息要求：

- 使用英文前缀 + 简短英文描述
- 一次 commit 只表达一个主要意图
- 优先写结果，不写过程

推荐示例：

- `feat: add recoverable cron-based article queue`
- `fix: reject duplicate insight titles from article analysis`
- `docs: document project workflow and agent rules`

不要做：

- 混合多个不相关改动到同一次提交
- 使用模糊信息，例如 `update`、`fix stuff`、`misc`
- 对已经 push 的共享历史随意改写

## 禁止事项

- 不要把真实 API key、`.dev.vars`、`.env` 提交到 Git
- 不要默认清空数据库后重跑全量数据
- 不要把原文英文内容直接当成最终中文页面内容
- 不要重新引入“人工审核后才能发布”的旧流程
- 不要采集超出 `a16z Articles / Investment News` 范围的数据

## 正常运行原则

正常运行应始终遵循：

- 增量抓取
- 数据落库
- 状态驱动处理
- 已发布内容直接从库中读取

除非明确在做一次性修复或历史重建，否则不要使用“清库重来”的方式处理日常任务。
