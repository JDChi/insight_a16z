# AGENTS.md

## 项目定位

这是一个部署在 `Cloudflare` 上的 `a16z AI 中文洞察站`。

目标不是做英文原文翻译站，而是：

- 收集 `a16z` 官网 `Articles` 与 `Investment News`
- 用中文输出结构化分析结果
- 聚合专题与周报
- 让前台直接消费已发布内容

中文标题代表本站的洞察标题，不是原文直译。英文原标题只作为来源信息保留。

## 当前技术结构

- `apps/web`: `Astro` 前台内容站和最小后台
- `apps/api`: `Hono + Cloudflare Workers` API、采集、分析、任务流
- `packages/core`: 共享 schema、类型、fixture
- `migrations`: `D1` 数据库初始化脚本

## 数据源边界

只允许采集：

- `a16z` 官网
- `Articles`
- `Investment News`

不要采集：

- `Podcast`
- `Video`
- 栏目页、导航页、落地页
- 其他媒体、基金或第三方网站

## 内容策略

- 页面主标题使用中文洞察标题
- 原文英文标题保留为来源信息
- 页面正文是本站分析结果，不是原文全文翻译
- 前端可见 UI 必须保持中文

## 发布与状态流转

文章状态以数据库为准。

当前约定状态：

- `ingested`: 已抓取入库，待分析
- `processing`: 正在分析
- `published`: 已发布到前台
- `failed`: 技术性失败，需人工排查

重要规则：

- 不再走人工审核前置流程
- 分析成功后直接发布
- 如果 AI 输出不合理，不要发布
- 不合理输出应回退到 `ingested`，等待后续重试
- 不要用劣质 fallback 结果冒充正常分析结果

## 队列与恢复策略

当前采用 `cron` 驱动的可恢复批处理模式，而不是内存长跑队列。

正确行为：

- 抓取先落库
- 后续由 `cron` 周期性扫描 `ingested`
- 每次只处理一小批
- 进程重启后，不依赖内存恢复
- 下次 `cron` 继续从数据库状态接着跑

不要重新引入：

- 单进程内存长跑队列
- 重启即丢状态的任务执行器

## 去重与质量约束

- 以 `source_url` 做文章级去重
- 中文标题不能与现有已发布标题撞车
- 如果生成结果与现有标题重复，应视为不合理输出并回退
- 不要为了“不断队列”而批量生成模板化、同质化标题

## 本地开发约定

- 本地敏感配置使用 `apps/api/.dev.vars`
- `.dev.vars`、`.env` 不允许提交
- `wrangler.toml` 可能包含本地 Cloudflare 绑定信息，修改前先确认是否应提交

本地常见模式：

- `apps/web` 可用 fixture 模式
- 联调时让前端走本地 API
- 真实模型 key 仅通过环境变量注入

## 测试约定

改动以下行为时，必须补或更新测试：

- 抓取规则
- AI 分析输出修复逻辑
- 状态流转
- 队列处理
- 后台接口语义

优先覆盖：

- 单元测试：解析、标题、防重、输出校验
- 集成测试：状态流转、发布、后台接口
- E2E：前台和后台核心页面

## 禁止事项

- 不要把真实 API key、`.dev.vars`、`.env` 提交到 Git
- 不要默认清空数据库后重跑全量数据
- 不要把原文英文内容直接当成最终中文页面内容
- 不要重新引入“人工审核后才能发布”的旧流程
- 不要采集超出 `a16z Articles / Investment News` 范围的数据

## 正常运行原则

正常运行应始终遵循：

- 增量抓取
- 数据落库
- 状态驱动处理
- 已发布内容直接从库中读取

除非明确在做一次性修复或历史重建，否则不要使用“清库重来”的方式处理日常任务。
